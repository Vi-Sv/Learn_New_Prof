## ЧАСТЬ 1: АНАТОМИЯ СЛОВАРЕЙ И ПРЕДОХРАНИТЕЛИ KEYERROR

Мы перешли от простых линейных очередей к сложным ассоциативным массивам. Данные в оперативной памяти теперь связаны жесткими парами «Ключ-Значение», имитируя строки и ячейки будущих ИТ-таблиц.

### Шаг 1. Создание карточки вагона в RAM

- **Код:**
  ```python
  vagon_card = {"number": 5408, "cargo": "зерно", "status": "простой"}
  ```
- **Физика процесса:** В оперативной памяти (RAM) зафиксирована новая структура данных — словарь (`dict`). Фигурные скобки `{}` указывают ядру Python на необходимость связать текстовые ключи (метки колонок) с конкретными физическими значениями под одним общим ярлыком `vagon_card`.

### Шаг 2. Извлечение атрибута по текстовому Ключу

- **Код:**
  ```python
  print(vagon_card["cargo"])
  ```
- **Физика процесса:** Вместо числовых индексов мы выполнили точечный вызов данных по имени ключа. Система проигнорировала порядок элементов, нашла в RAM метку `"cargo"` и мгновенно вывела на верстак привязанное к ней значение `"зерно"`.

### Шаг 3. Столкновение с инфраструктурным капканом отсутствия данных

- **Код:**
  ```python
  print(vagon_card["weight"])
  ```
- **Физика процесса:** Скрипт упал с критической ошибкой `KeyError: 'weight'`. Движок Python не обнаружил указанной метки среди зарегистрированных ключей словаря и выдал аварийный стоп-сигнал. На живом производстве такой сбой намертво останавливает автоматические конвейеры, если поставщик изменил состав полей.

### Шаг 4. Безопасный перехват через метод `.get()`

- **Код:**
  ```python
  safe_weight = vagon_card.get("weight", 0)
  print(safe_weight)
  ```
- **Физика процесса:** Метод `.get()` выступил в роли встроенного предохранителя (аналог `COALESCE` в SQL). Система мягко проверила наличие ключа `"weight"`. Не найдя его, она заблокировала вылет ошибки `KeyError` и хладнокровно вернула указанное через запятую резервное значение — числовой ноль `0`.

## ЧАСТЬ 2: ПРОЕКТИРОВАНИЕ ТАБЛИЧНЫХ МАТРИЦ И БИЗНЕС-ФИЛЬТРАЦИЯ

Мы объединили списки `[]` и словари `{}` в единую многомерную структуру — список словарей. Это точный технический прообраз полноценной ИТ-таблицы, где внешние элементы определяют строки, а внутренние — колонки.

### Шаг 5. Формирование цифрового состава

- **Код:**
  ```python
  vagon_train = [
      {"number": 5408, "cargo": "зерно", "status": "простой"},
      {"number": 4441, "cargo": "щебень", "status": "в пути"},
      {"number": 3372, "cargo": "зерно", "status": "простой"}
  ]
  ```
- **Физика процесса:** В RAM запечатан многомерный массив. Внешний контур (квадратные скобки) выделил память под строки-списки, а внутри каждой ячейки расположился отдельный независимый словарь со своим набором бизнес-параметров.

### Шаг 6. Сквозной парсинг колонок через цикл `for`

- **Код:**
  ```python
  for vagon in vagon_train:
      print(vagon["number"])
  ```
- **Физика процесса:** Автоматический конвейер `for` прошагал по строкам списка `vagon_train`. На каждой итерации (круге) система брала очередной внутренний словарь, вешала на него временный ярлык `vagon`, а внутренняя команда по ключу `["number"]` извлекала и печатала бортовые номера.

### Шаг 7. Бизнес-фильтрация по условию простоя (Реализация блока WHERE)

- **Код:**
  ```python
  delay_vagons = []

  for vagon in vagon_train:
      if vagon["status"] == "простой":
          delay_vagons.append(vagon["number"])

  print(delay_vagons)
  ```
- **Логика конструкции:** Сито условий `if` на каждом круге заходило в колонку-ключ `["status"]` и проводило логическое сравнение через оператор `==`. Если условие возвращало истину, метод `.append()` забирал бортовой номер вагона и накапливал его в изначально пустую чистую корзину `delay_vagons`.
- **Физика процесса:** На втором круге (вагон `4441`) система прочитала статус `"в пути"`, сравнение вернуло ложь, и конвейер отбросил этот элемент как лишний информационный шум. На верстак выгружен стерильный аналитический инсайт — массив только тех вагонов, которые генерируют убытки: `[5408, 3372]`.
